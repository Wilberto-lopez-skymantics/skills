# Stage 01b: Adopt Existing Codebase into SDD

Reverse-engineer specs from a working codebase тЖТ bring project under SDD management.

This stage is an **alternative to Stage 01 (Brainstorming)**. It produces the same output artifacts but extracts them from existing code rather than designing them from scratch.

## When to Use This Stage (instead of Stage 01)

- The project has working code but no `specs/SPEC.md`
- The user wants to bring an existing codebase under SDD management
- The user says "adopt", "onboard", or "reverse-engineer the spec"

## Inputs

- Layer 4 (working): Existing codebase (source files, config files, package manifests)
- Layer 4 (working): Existing tests (if any)
- Layer 4 (working): Existing documentation (README, inline comments, git history)
- Layer 3 (reference): `../_config/decision-log-template.md`
- Layer 3 (reference): `../_config/design-template.md` (UI only)
- Layer 3 (reference): `../_config/references/confidence-assessment.md`

## Process

### Step 1: Configure Adoption Parameters

Present adoption parameters to user and get confirmation before scanning:

| Parameter | Default | Description |
|-----------|---------|-------------|
| Analysis Depth | `thorough` | `quick` / `standard` / `thorough` |
| Focus Modules | `all` | Directories/modules to prioritize |
| Include Git History | `yes` | Mine git log for decisions |
| Map Existing Tests | `yes` | Inventory tests & map to invariants |
| Project Type | auto-detect | `client-side-app` / `server-side-app` / `library` / `cli-tool` / `full-stack` / `extension` |

**тЫФ GATE:** тКе begin analysis until user confirms parameters.

### Step 2: Scan & Inventory

1. Read folder tree, package manifests, entry points, README
2. Identify key modules, responsibilities, public APIs, dependencies
3. If `Map Existing Tests = yes`: read test files, categorize (unit/integration/e2e), note coverage
4. If `Include Git History = yes`: mine git log for architectural decisions & evolution

### Step 3: Confidence Self-Assessment

Score understanding across 5 dimensions. Read `../_config/references/confidence-assessment.md`.

! reach тЙе85 before proceeding. If below тЖТ ask user targeted questions about weak dimensions.

**тЫФ GATE:** тКе draft artifacts until Confidence score тЙе85.

### Step 4: Draft Spec Artifacts

Draft in order:
1. `specs/DECISION_LOG.md` тАФ discovered decisions. Mark inferred decisions with `[inferred]`. Use `../_config/decision-log-template.md`.
2. `specs/SPEC.md` тАФ reverse-engineered requirements. тКе aspirational. Document what code does today. Mark technical debt as `[TECH DEBT]`.
3. `specs/DESIGN.md` (UI only) тАФ extracted design tokens from CSS/styles. Use `../_config/design-template.md`.
4. Test-to-invariant mapping тАФ link existing tests тЖТ spec invariants. Flag `[UNMAPPED]` tests & `[UNTESTED]` invariants.
5. `specs/IMPLEMENTATION_PHASES.md` тАФ all phases `[x]` (completed). Must include adoption header:
   > **Note:** Auto-generated by SDD Adoption. All phases pre-checked (existing code). Future changes add new phases via standard SDD lifecycle.

### Step 5: User Review

Present all drafted artifacts. User knows the codebase better тЖТ they catch misinterpretations.

**тЫФ GATE:** тКе proceed until user explicitly approves adopted spec artifacts.

### Step 6: Set State & Transition

Write `specs/.sdd-state.json` (schema: `specs/.sdd/_config/sdd-state.json.template`):
```json
{
  "lastCompletedStep": 1,
  "stepName": "Adoption Complete",
  "projectType": "[detected]",
  "timestamp": "[ISO 8601]",
  "notes": "Project adopted into SDD. Spec reverse-engineered from existing codebase."
}
```

Present next-step options to user:
- **Option A: Run Adversarial Analysis** тЖТ Stage 02 (harden adopted spec, find gaps)
- **Option B: Expand via Brainstorming** тЖТ Stage 01 (use adopted spec as baseline, brainstorm new features)
- **Option C: Jump to Development** тЖТ Stage 04/05 (SDD for future changes only)

Prompt the user to choose, then transition.

## Outputs

- `specs/SPEC.md`
- `specs/DECISION_LOG.md`
- `specs/DESIGN.md` (if UI project)
- `specs/IMPLEMENTATION_PHASES.md` (pre-checked)
- `specs/.sdd-state.json`
- `specs/backprop-log.md` (created empty if not present)
